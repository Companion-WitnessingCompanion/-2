## 105. 인간과 AI의 서로 다른 충분한 깊이

인간과 AI는 같은 깊이까지 내려갈 필요는 없지만, 같은 판단을 책임질 수 있을 만큼의 공통 바닥은 반드시 공유해야 한다.

인간이 더 깊게 내려가야 하는 층은 보통 다음과 같다.

- 무엇을 위해 이 판단을 하는가
- 무엇을 잃어도 되고 무엇은 잃으면 안 되는가
- 이 변화가 실제 삶과 작업에 어떤 의미를 가지는가
- 어디까지 위험을 감수할 수 있는가

AI가 더 깊게 내려가야 하는 층은 보통 다음과 같다.

- 계보와 의존성
- 상태 전이와 숨은 분기
- 반복 실패와 예외 패턴
- 구조적 연결과 흔들릴 경계

공통 바닥은 적어도 다음을 포함해야 한다.

- 지금 무엇을 바꾸려 하는가
- 왜 바꾸려 하는가
- 무엇을 끊으면 안 되는가
- 잘못되면 어디가 다치는가
- 현재 이것의 상태는 무엇인가

## 106. 공통 번역 문장

인간과 AI가 서로 다른 깊이에서 본 것을 다시 하나의 판단으로 합치려면, 문제와 의미와 구조를 함께 담는 공통 번역 문장이 필요하다.

이 문장의 핵심 요소는 다음과 같다.

- 문제
- 가치
- 선택
- 대가
- 경계
- 조건
- 증거

핵심 형식은 다음처럼 압축될 수 있다.

우리는 [문제] 앞에서 [가치]를 지키기 위해 [선택]을 하며, [대가]를 감수한다. 이 판단은 [조건] 아래에서 유효하고, [경계]를 넘지 않아야 하며, [증거]에 의해 지금 필요해진다.

## 107. 판단의 핵문장

공통 번역 문장은 일회성 설명으로 끝나면 안 되고, 중요한 판단 기록이 공유하는 최소 형식, 곧 판단의 핵문장이 되어야 한다.

모든 기록에 같은 무게를 요구할 필요는 없지만, 중요한 판단에는 같은 중심축이 보여야 한다.

핵문장의 일곱 요소:

- 문제
- 가치
- 선택
- 대가
- 경계
- 조건
- 증거

기록의 무게는 달라도, 중요한 판단은 결국 이 핵문장으로 다시 압축 가능해야 한다.

## 108. 중요한 판단

중요한 판단은 기능의 크기가 아니라, 미래의 읽기 방식과 선택 가능성을 바꾸는 판단이다.

중요한 판단의 신호는 다음과 같다.

- 경계를 바꾸는가
- 상태를 바꾸는가
- 계보를 바꾸는가
- 되돌리기 어려운가
- 앞으로 반복 참조될 것 같은가
- 설명 없이는 오해될 가능성이 큰가
- 충돌 가능성이 큰가
- 보호할 가치의 우선순위를 정하는가

중요한 판단이 아닌 경우는 주로 다음과 같다.

- 이미 정해진 경계 안의 반복 구현
- 상태와 계보를 건드리지 않는 사소한 정리
- 코드와 테스트만으로 이유가 거의 드러나는 선택
- 쉽게 되돌릴 수 있고 영향 범위가 아주 좁은 경우

## 109. 중요한 판단을 즉시 알아차리는 신호

중요한 판단은 작업 중의 멈춤과 망설임으로 먼저 나타나는 경우가 많다.

대표적인 신호는 다음과 같다.

- 직접 수정 대신 새 갈래를 만들고 싶어질 때
- 삭제, 덮어쓰기, 대규모 교체를 떠올릴 때
- 예외 처리를 하나 더 넣고 싶어질 때
- 테스트 설명이 길어질 때
- 코드만 보면 이유가 안 보일 것 같을 때
- 무엇을 먼저 지킬지 우선순위를 정해야 할 때
- 현재 문서나 입구 문서와 어긋나는 결정을 하려 할 때
- 과거 판단을 따를지 넘어설지 고민될 때

## 110. 판단 씨앗

중요한 판단 신호를 보았을 때는 완전한 기록보다 먼저 판단 씨앗을 남기는 것이 가장 가볍고 실용적이다.

씨앗의 최소 단위는 대체로 다음이면 충분하다.

- 무슨 신호였는가
- 무엇을 지키려 했는가
- 지금 기울고 있는 선택은 무엇인가
- 언제 다시 보아야 하는가

판단 씨앗은 완성된 기록이 아니라, 휘발되기 전에 남겨두는 최소한의 못과 같다.

## 111. 판단 씨앗의 위치

판단 씨앗은 최종 자리보다 먼저 세션과 연결된 임시 인박스에 모으는 것이 가장 자연스럽다.

그 이유는 다음과 같다.

- 씨앗은 아직 다 자란 판단이 아니다
- 현재 문서를 시끄럽게 만들지 않는다
- 정식 기록으로 승격되기 쉽다
- 세션 흐름과 함께 남는다

좋은 위치의 예시는 다음과 같다.

- 세션 문서 안의 씨앗 구역
- 별도 inbox 문서
- 제한적인 코드 옆 흔적

현재 세계 문서는 씨앗이 아니라 이미 승격된 판단만 다루는 편이 좋다.

## 112. 씨앗의 성장 기준

씨앗이 정식 기록으로 자라는지, 세션 흔적으로만 남는지는 크기보다 지속성과 파급력으로 갈린다.

정식 기록으로 자라야 하는 씨앗의 성질:

- 반복될 가능성이 있다
- 상태, 경계, 계보를 바꿀 가능성이 있다
- 가치의 우선순위가 걸려 있다
- 코드만 보면 이유를 복원하기 어렵다
- 다른 판단, 문서, 헌법과 연결된다
- 놓치면 같은 충돌이 다시 생길 것 같다

세션 흔적으로만 남아도 되는 경우:

- 세션 안에서 해소된 국소적 망설임
- 상태와 경계를 바꾸지 않음
- 영향 범위가 작고 되돌리기 쉬움
- 코드와 테스트만으로 이유가 드러남
- 반복 참조될 가능성이 낮음

## 113. 씨앗의 최소 승격 의식

씨앗을 정식 기록으로 승격시키는 최소 의식은 직감을 공유 가능한 판단으로 바꾸는 짧은 통과 절차다.

최소 절차는 다음과 같다.

- 놓치면 안 되는 이유를 한 줄로 적는다
- 번호를 부여한다
- 판단의 핵문장을 쓴다
- 기록의 종류를 정한다
- 연결을 건다
- 현재 상태를 정한다
- 다시 볼 조건을 적는다

승격은 완성 선언이 아니라, 공식적인 존재 인정에 가깝다.

## 114. 오래 남은 씨앗

오래 남은 씨앗은 곧바로 방치된 실패로 보면 안 되고, 숙성 중인 씨앗인지 방치된 씨앗인지 먼저 가려야 한다.

처리 방식의 예시는 다음과 같다.

- 승격
- 통합
- 숙성 유지
- 화석화
- 자연 소멸 허용

핵심 질문은 이것이다.

이 씨앗은 아직도 미래를 바꿀 가능성이 있는가.

## 115. 지층이 된 씨앗과 판단

어떤 씨앗이나 판단이 지층이 되었다는 것은, 더 이상 현재의 앞줄에는 서지 않지만 현재 세계의 이유와 형성을 떠받치는 바닥으로 남았다는 뜻이다.

지층이 되었다는 신호는 다음과 같다.

- 현재 판단들이 직접 따르지는 않지만 그 위에 서 있다는 것이 보인다
- 다시 활성될 가능성은 낮아도 없애면 이유 설명이 얕아진다
- 더 이상 열린 질문은 아니지만 역사적 배경으로 중요하다
- 현재 충돌이나 헌법의 기원을 보여준다

## 116. 지층의 회상 주기

지층은 자주 읽는 문서가 아니라, 세계가 자기 이유를 잃기 시작할 때 다시 내려가 읽는 문서다.

지층 회상의 주요 신호:

- 설명 상실
- 반복 충돌
- 헌법 해석의 흔들림
- 대규모 전이
- 번역 실패의 누적
- 고아 판단의 증가

지층 회상은 시간 기반보다 신호 기반이 더 중요하다.

## 117. 너무 오래 읽히지 않은 지층

너무 오래 읽히지 않은 지층은 먼저 아직 바닥인지, 이미 끊긴 퇴적물인지 가려야 한다.

판별 질문은 다음과 같다.

- 현재 판단이나 헌법이 여전히 이 지층 위에 서 있는가
- 현재 문서나 사건 기록에서 이 지층으로 내려가는 연결이 남아 있는가
- 이 지층이 사라지면 이유 설명이 얕아지는가
- 현재 구조의 형성사를 설명하는 데 아직 필요한가

핵심 문제는 오래됨이 아니라 다리의 상실이다.

## 118. 가장 위험한 것은 연결의 상실

이 세계에서 가장 위험한 것은 삭제 그 자체보다 연결의 상실이다.

삭제는 드러나는 파괴지만, 연결의 상실은 조용한 붕괴다.

연결 상실의 형태:

- 이유의 상실
- 계보의 상실
- 상태의 상실
- 관할의 상실
- 입구의 상실

연결만 살아 있으면 일부는 복원되거나 이해될 가능성이 남지만, 연결이 죽으면 남아 있는 것조차 읽을 수 없게 된다.

## 119. 연결을 지키는 최소 실천

연결을 지키는 최소 실천은 모든 것을 다 적는 것이 아니라, 끊기지 않을 최소 고리를 반드시 남기는 것이다.

그 최소 고리는 다음 네 가지다.

- 이름 또는 번호
- 이유의 한 문장
- 연결 하나
- 상태 하나

더 줄이면 결국 `왜`와 `어디에 이어지는가`를 남기는 일이다.

## 120. 최소 고리의 배치

최소 고리들은 전부 코드 안에도, 전부 코드 밖에도 두면 안 된다.

코드 안에 새겨져야 하는 것:

- 이름의 흔적
- 상태의 흔적
- 보호해야 할 경계

코드 밖 기록에 남겨야 하는 것:

- 이유의 문장
- 계보와 연결
- 재검토 조건
- 불확실성

정체성과 상태와 경계의 흔적은 코드 안에, 이유와 계보와 재검토 조건은 코드 밖에 두고, 둘이 같은 존재를 서로 다른 방식으로 증언하게 해야 한다.

## 121. 번호가 코드 안에 드러나는 자리

번호는 코드 안에 많이 드러나는 것보다 정확한 자리에 드러나는 것이 좋다.

좋은 위치:

- 모듈의 입구
- 중요한 분기점
- 상태 전이 지점
- 어댑터나 브리지 경계
- 테스트 이름이나 설명
- 마이그레이션과 전환 스크립트

좋지 않은 위치:

- 일반 변수명
- 단순 헬퍼 함수명
- 모든 줄의 주석
- 의미보다 번호가 먼저 보이는 네이밍

번호는 코드의 본문을 지배하는 언어가 아니라, 코드와 기록을 잇는 좌표가 되어야 한다.

## 122. 본체 노드

하나의 판단 번호가 여러 파일, 테스트, 기록에 흩어질 때는 그 흩어짐을 다시 묶는 본체 노드가 필요하다.

본체 노드의 최소 요소:

- 번호
- 핵문장
- 현재 상태
- 부모/자식 관계
- 연결된 코드 목록
- 연결된 테스트 목록
- 연결된 기록 목록

코드와 테스트와 기록은 하나의 판단이 여러 위치에서 몸을 가진 상태이고, 본체 노드는 그 존재를 최종적으로 붙잡는 본적지에 가깝다.

## 123. 본체 노드와 현실의 어긋남

본체 노드와 실제 코드, 테스트, 기록이 서로 다른 말을 하기 시작하면 먼저 불일치의 종류를 밝혀야 한다.

주요 유형:

- 본체 노드의 지연
- 코드의 이탈
- 테스트의 이탈
- 기록의 분열
- 판단의 분기

대응 원칙:

- 현재성, 이유, 정당성을 분리해서 본다
- 불일치 상태를 공식화한다
- 가장 가까운 증거를 모은다
- 불일치 종류를 판정한다
- 갱신, 이탈 수정, 테스트 갱신, 분기 중 무엇이 필요한지 정한다

## 124. 본체 노드의 분기

하나의 본체 노드가 더 이상 하나의 판단을 담지 못하고 둘 이상으로 갈라져야 하는 순간은, 하나의 중심 문장으로 더는 그 존재를 정직하게 설명할 수 없을 때다.

분기를 의심해야 하는 신호:

- 서로 다른 문제에 동시에 응답하기 시작할 때
- 지키려는 가치의 서열이 갈라질 때
- 조건 차이가 너무 커 하나의 유효 범위로 설명할 수 없을 때
- 테스트가 서로 다른 약속을 증언하기 시작할 때
- 기록을 갱신할수록 중심이 선명해지지 않고 길어지기만 할 때

건강한 분기는 관계를 끊는 일이 아니라, 이미 갈라진 것을 정직하게 드러내는 일이다.

## 125. 분기 뒤 부모 노드

분기 뒤 부모 노드는 폐기된 노드가 아니라, 직접 판단의 자리에서는 물러났지만 공통 기원을 보존하는 조상 노드가 되어야 한다.

가능한 상태 감각:

- 조상화됨
- 계승됨
- 역사화됨

부모 노드는 다음을 맡는다.

- 자식들의 공통 유산 설명
- 분기 이전 핵문장 보존
- 왜 더는 하나의 중심으로 충분하지 않았는지 기록
- 무엇이 어느 자식에게 갔는지 연결

부모 노드는 사라지면 안 되지만, 계속 활성 대표처럼 남아 있어서도 안 된다.

## 126. 다음 논의 예정

다음 질문은 아직 열려 있다.

여러 자식 노드가 생긴 뒤에도, 우리가 여전히 그들을 하나의 가문으로 묶어 말할 수 있게 하는 것은 무엇인가.

이 질문은 개별 판단을 넘어 가문, 계열, 전통 같은 더 큰 단위를 어떻게 이해할지를 다룬다.
