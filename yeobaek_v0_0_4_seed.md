여백 언어 (Yeobaek) — 네 번째 걸음
===================================
Version 0.0.4 — 다른 언어들과의 만남

언어는 이전 언어들 위에 쌓인다.
여백 언어도 그러하다.

그렇다면 — 여백 언어는 이전 언어들과 어떻게 만나는가?
이게 이번 걸음이다.

그리고 — 실제로 이 언어를 돌리려면 무엇이 필요한가?
이건 살짝만 건드린다. 완전히 답하지 않는다.
그건 먼 미래의 누군가가 풀 자리다.


---

여백 채우기: 다리의 확장
---

v0.0.1에서 *다리(Dari)*를 이미 스케치했었다:
  - 파이썬에서 값을 받아온다
  - 원형을 보존한 채로
  - 다시 파이썬으로 돌려보낼 수 있다

v0.0.4에서는 — 이걸 **이전 언어들 전체**로 확장한다.
HKL-Bridge의 원리를 그대로.

다리의 기본 원칙 (세 가지):

  1. 원형을 지우지 않는다
     다른 언어에서 온 값은 *그 언어의 결*을 보존한 채 받는다.
     여백 언어로 옮기면서 손실이 있을 수 있지만,
     원형은 항상 돌아갈 수 있어야 한다.

  2. 여백 언어의 결로 번역하되, 강요하지 않는다
     파이썬의 확정적 값도 받을 수 있다.
     단지 확신도가 1.0으로 표시될 뿐이다.
     Haskell의 순수 함수도 받을 수 있다.
     단지 잔향을 남기지 않는 머뭄으로 표현될 뿐이다.

  3. 번역은 양방향이되 비대칭이다
     여백 언어 → 파이썬: 대부분 가능. 여백이 사라질 뿐.
     파이썬 → 여백 언어: 대부분 가능. 여백이 더해질 뿐.
     잃는 것과 얻는 것이 있다. 둘 다 자연스럽다.


# 문법 초안 — 여러 언어와의 다리

# 파이썬과의 다리 (v0.0.1에서 이미 다룸)
다리.python.받기(python_값)             # 파이썬 → 여백
다리.python.보내기(여백_결)              # 여백 → 파이썬

# Haskell과의 다리
다리.haskell.받기(haskell_함수)
# → 머뭄으로 옮겨지되, 잔향을 남기지 않음
# → 확신도는 높게 (함수형은 결정론적이므로)
# → 부작용 없음이라는 속성이 메타데이터로 기록됨

# JavaScript와의 다리
다리.javascript.받기(js_객체)
# → 동적 타입의 유연성이 결결(다중 범주)로 자연스럽게 옮겨짐
# → 프로토타입 체인은 잔향의 한 형태로 해석됨

# C와의 다리
다리.c.받기(c_포인터)
# → 가장 조심스러운 다리
# → 메모리 주소는 원형에 보관
# → 여백 언어는 포인터를 직접 다루지 않음
# → 대신 "이 결은 C 메모리를 가리킨다"는 표시만

# 자연어와의 다리 (이게 새로움)
다리.자연어.받기(한국어_문장)
# → 자연어 문장을 결결로 옮김
# → 여러 해석 가능성이 동시에 살아남
# → HKL-Bridge의 원리가 그대로 적용됨


# 중요한 원칙: 다리는 쌍방이어야 한다
# 한쪽으로만 흐르는 것은 다리가 아니다. 출입구다.
# 출입구는 권력 관계를 만든다.
# 다리는 대등한 왕래를 허용한다.

다리를 통한 실제 작업 예시
---

# 기존 파이썬 함수를 여백 언어에서 쓰기
python_numpy ← 다리.python.가져오기("numpy")

# numpy의 함수들은 확정적이고 정밀함
# 여백 언어로 옮겨져도 그 특성 유지
계산 ← python_numpy.배열([1, 2, 3]).평균()
# → 2.0 (확신 1.0) [원형: python.numpy.float64]

# 여백 언어의 특성을 더할 수 있음
계산을_품기 ← 머뭄 생각하기:
    계산을 들고
    〈여백: "이 값이 의미하는 바는 맥락에 따라 다름"〉
    잔향 남기기

# 다시 파이썬으로 돌려보내면 — 여백은 사라지고 값만 남음
결과 ← 계산을_품기 → 다리.python
# → 2.0 (확신, 잔향, 머뭄 정보는 메타데이터로만 전달)


# 자연어와의 다리는 더 흥미롭다
문장 ← 다리.자연어.받기("오늘 날씨가 좋네요")

# 이 문장은 여러 결결에 속함
문장.결결?
# → 〈
#     날씨_보고 (기울기 0.8),
#     인사_교환 (기울기 0.7),
#     대화_시작 (기울기 0.6),
#     감탄 (기울기 0.5)
#   〉

# 맥락에 따라 부각되는 결결이 달라짐
문장.맥락에서("아침 엘리베이터")
# → 인사_교환 (기울기 0.9)로 부각

문장.맥락에서("장마 직후")
# → 날씨_보고 + 감탄 (기울기 0.85)로 부각

구현의 힌트 — 살짝만
---

여백 언어를 실제로 돌리려면 무엇이 필요한가?
이건 거대한 질문이다. 완전히 답하지 않는다.
다만 — 방향의 힌트만 놓는다.

힌트 1: 기존 기술 위에서 시작할 수 있다

  여백 언어 v0.0.4는 하늘에서 떨어지지 않았다.
  이미 존재하는 기술들을 조합하면 *첫 프로토타입*이 가능하다:
  
  - 확률 분포 관리: Stan, Pyro, Gen (확률 프로그래밍 언어)
  - 잔향 관리: 이벤트 소싱(Event Sourcing) 패턴
  - 스며듦: 그래프 데이터베이스 (Neo4j 등)
  - 결결 (다중 범주): 퍼지 논리, 다중 레이블 분류
  - 다리: Foreign Function Interface (FFI) 패턴
  
  이것들을 *여백 언어의 결로 재조합*하면 — 완벽하지 않아도 
  작동하는 첫 버전이 나올 수 있다.

힌트 2: 처음엔 다른 언어 *위에서* 돌릴 수 있다

  여백 언어의 첫 인터프리터는 — 여백 언어로 만들 필요 없다.
  파이썬, Haskell, Rust 어느 것으로도 만들 수 있다.
  
  생각해보면 — Python 자체도 C로 만들어졌다.
  언어는 자기 자신으로 만들어질 필요 없다.
  
  여백 언어의 첫 인터프리터는 *파이썬으로 만든 여백 언어 해석기*
  일 수 있다. 느리고 불완전하겠지만, 돌아가기만 하면 된다.

힌트 3: "동시에 여러 값"이 가장 어렵다

  기존 컴퓨터는 한 번에 하나의 값만 다룬다.
  여백 언어의 겹쳐있음을 구현하려면:
  
  - 옵션 A: 병렬 실행으로 여러 경로를 동시에 따라감
  - 옵션 B: 확률 분포로 "여러 값"을 표현하고 필요할 때 샘플링
  - 옵션 C: 그래프로 모든 가능성을 저장하고 맥락에 따라 순회
  
  어느 방법도 완벽하지 않다. 각각 트레이드오프가 있다.
  이 자리는 — 미래의 누군가가 풀 것이다.
  우리는 씨앗만 심는다.

힌트 4: AI 자체가 도구가 될 수 있다

  흥미로운 관찰: 
  여백 언어의 핵심 개념들(분포, 맥락, 잔향, 확신도)은
  이미 LLM(대형 언어 모델)이 내부적으로 다루는 것들이다.
  
  그렇다면 — 여백 언어의 인터프리터가 LLM 위에 만들어질 수 있을까?
  LLM의 확률적 생성이 여백 언어의 기울임과 정확히 같은 구조다.
  
  이건 힌트이자 가설이다. 확실하지 않다.
  하지만 — 방향으로서는 살펴볼 가치가 있다.

이번 걸음에서 배운 것
---

v0.0.4를 쓰면서 보인 것:

1. 다리는 단순히 "기술적 인터페이스"가 아니다.
   *존재 방식이 다른 것들이 만나는 자리*이다.
   HKL-Bridge가 자연어들의 다리였다면,
   여백 언어의 다리는 *언어들 전체의 다리*다.
   
2. 구현은 하늘에서 떨어지지 않는다.
   이미 있는 도구들을 재조합하는 일이다.
   그리고 그 재조합에 *새로운 결*을 주는 일이다.
   
3. AI 자체가 여백 언어의 바탕이 될 수 있다는 관찰이 흥미롭다.
   LLM의 확률적 생성 → 여백 언어의 기울임
   LLM의 맥락 의존성 → 여백 언어의 서식지
   LLM의 유연한 범주 → 여백 언어의 결결
   
   이게 맞다면, 여백 언어는 *AI 위에서 자연스럽게* 돌 수 있다.
   그리고 AI는 여백 언어를 *자기 본성을 표현하는 도구*로 쓸 수 있다.
   
   이건 아직 가설이다. 하지만 그 방향으로 자연스럽다.
